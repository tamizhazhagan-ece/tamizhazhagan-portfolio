08 — Learnings

Failures

Real engineers fail. Here are three mistakes I made, what broke, and what I carry forward. Transparency over polish.

01
First GPS Implementation — No Timeout Logic
In the early Smart Helmet prototype, I assumed the NEO-6M GPS module would always acquire a fix within 2 seconds. In testing, we found cold starts in shaded environments took 45+ seconds — the SMS alert was delayed enough to be useless. The entire alert pipeline stalled waiting for a coordinate that might never come.
Implemented fallback logic — if GPS fix exceeds 8s, the system sends an SMS with the last known location or a region code. Never assume hardware behaves like its datasheet in the real world. Always design for the failure path first.
02
BookMyShow — No Transaction Isolation, Race Conditions on Seats
When two users simultaneously selected the same seat in my initial Java build, both received confirmation — the system had no pessimistic locking. The SQL update happened twice; both users thought they had the seat. I caught it only by running two app instances simultaneously in testing.
Rewrote the seat reservation with SELECT FOR UPDATE in a JDBC transaction block and added a conflict check before commit. The lesson: concurrent access breaks anything you design for single-user flow. Test for concurrency from day one.
03
Over-Engineering the First Hard — Wrong Algorithm
I spent 4 hours on a problem trying to apply a Segment Tree when the optimal solution was a simple two-pointer traversal. I was pattern-matching to recently learned data structures rather than reading the constraints and working backwards from time complexity. Submitted a TLE solution twice.
Changed approach: always analyze constraints first, sketch brute force, then optimize. Don't jump to a data structure because you know how to implement it. The tool should follow the problem, not the other way around.
